# Cluster Context-Hub

This skill encodes a five-phase agent-driven clustering workflow.

```
Phase 1  Discovery        →  read sample, propose 12-18 categories
Phase 2  Assignment       →  dispatch N parallel subagents (one per batch)
Phase 3  Aggregate        →  merge batch results, audit distribution
Phase 4  Sub-cluster      →  split any cluster > 30% (optional, recursive)
Phase 5  Scope authoring  →  write description + routing_scope per Auto/*
```

The output is `<hub_root>/_index/synthetic_tags.yaml` (path → Auto path) plus
new entries in `<hub_root>/_index/scopes.yaml`. The user's frontmatter tags
are NEVER touched. Every overlaid note appears in BOTH its user-tag positions
AND its single agent-cluster path — that's the dual-track rendering.

## When to use

- The user has a context-hub at `<hub_root>` (e.g. `~/context-hub`, or
  wherever `context-hub import` wrote to), and wants topic-driven hierarchy
  generated by the agent.
- A previous `synthetic_tags.yaml` exists but is stale, too coarse, or only
  covers untagged notes — the user wants a full-corpus refresh.
- The user wants the agent (you) to be the cluster-er, not embeddings or
  external models or hand-written rules.

## Key idea

Every classification decision is an agent decision: the agent literally reads
the note's body + tags and decides. This is enabled by:

1. **Phase 1 fixes the category space** so all later agent calls speak the
   same vocabulary. No drift.
2. **Phase 2 parallelizes** over batches via the `Task` tool — N subagents
   run concurrently, each handling ~150 notes.
3. **Phase 4 recurses** into oversized buckets the same way.

Result: every label is the product of an agent reading the note, but the
total cost is bounded (≈ ⌈corpus_size / 150⌉ + a few subagents).

## Workflow

> Throughout, `<HUB_ROOT>` is the hub root passed by the user (or the default
> `~/context-hub`).

### Phase 0: Pre-flight

Run `scripts/inspect_hub.py --root <HUB_ROOT>`. Look at:

- Total note count (affects batch count)
- Top tags (Phase 1 should respect / build on these)
- Length distribution (informs Phase 2 body cap)
- Existing `synthetic_tags.yaml` state (if any)

If notebook count is < 200 you can skip Phase 2's batching and label
directly. Otherwise proceed.

### Phase 1: Discovery (you do this directly, no subagent)

1. Sample:
   ```bash
   python scripts/sample_corpus.py --root <HUB_ROOT> \
     --out <HUB_ROOT>/_index/cluster-work/sample.md --n 160
   ```
2. Read `<HUB_ROOT>/_index/cluster-work/sample.md` end-to-end.
3. Follow `references/phase1_discovery.md` to derive 12-18 categories.
4. Write `<HUB_ROOT>/_index/cluster-work/categories.yaml` with the
   structure specified in that reference. Every category has `name`,
   `one_liner`, `typical_signals`, `example_paths`.

The `categories.yaml` is the contract every subagent will obey in Phase 2.

### Phase 2: Assignment (parallel subagent dispatch)

1. Batch:
   ```bash
   python scripts/make_batches.py --root <HUB_ROOT> --size 150
   ```
   Writes `batch-NN.md` files under
   `<HUB_ROOT>/_index/cluster-work/batches/`. Reports the number of batches N.

2. Dispatch N Task subagents IN A SINGLE MESSAGE, each with the prompt
   template from `references/phase2_assignment.md` (substitute `<HUB_ROOT>`,
   `<BATCH_FILE>`, `<RESULT_FILE>`). Subagent type: `general-purpose`.
   Run all of them with `run_in_background=true`.

3. As each subagent finishes (you receive completion notifications), no
   action needed until all are done.

### Phase 3: Aggregate + audit

```bash
mkdir -p <HUB_ROOT>/_index/cluster-work/results   # if not auto-created
python scripts/aggregate.py --root <HUB_ROOT>
python scripts/audit_distribution.py --root <HUB_ROOT>
```

The audit flags:
- Any cluster > 30% of corpus → Phase 4 candidate
- Any cluster < 2% → consider merging via hand-edit or accepting
- `Auto/未分类` > 5% → categories were too narrow; revisit Phase 1

If audit looks bad, you can either:
- Edit `categories.yaml` and re-dispatch Phase 2 (full rerun)
- Hand-edit `synthetic_tags.yaml` to merge/split a few notes (cheap fix)

### Phase 4 (optional): Sub-cluster oversized buckets

Follow `references/phase4_subcluster.md` for each `Auto/<top>` flagged
oversized. The end state replaces those `Auto/<top>` assignments with
`Auto/<top>/<sub>` in `synthetic_tags.yaml`.

### Phase 5: Scope authoring

```bash
python scripts/scaffold_scopes.py --root <HUB_ROOT>
```

Lists every `Auto/<path>` (top + sub-cluster) still missing a scopes entry,
with 3-5 sample notes per missing path. Follow
`references/phase5_scope.md` to author each entry. Append to (NOT
overwrite) `<HUB_ROOT>/_index/scopes.yaml` under `scopes:`.

### Phase 6: Render

```bash
context-hub reindex --root <HUB_ROOT>
```

Open `<HUB_ROOT>/_index/INDEX.md`. User-tag tree at top, `Auto/` tree at
bottom. Each note appears in both.

## Anti-patterns

- **Rule-based shortcut.** If you find yourself writing
  `if "keyword" in body`, you've fallen back to rule matching. The whole
  point of this skill is the agent IS the classifier. The only deterministic
  Python is sampling, batching, and aggregating — never the label decision.
- **Inventing categories in Phase 2.** Every Phase 2 subagent must use
  ONLY the categories from `categories.yaml`. If they need to invent, that
  signals Phase 1 was incomplete — revisit Phase 1, not Phase 2.
- **First-person scope text.** `description` is shown to users; no "用户
  认为" or "你倾向" allowed there. See `references/phase5_scope.md`.
- **Skipping Phase 0 inspect.** Without the corpus profile, you can't size
  batches sensibly. Always inspect first.

## Failure recovery

- A Phase 2 subagent times out / errors: just re-dispatch that one batch.
  `aggregate.py` is idempotent over result files.
- `aggregate.py` reports duplicates: the same path got two different labels
  from two batches — shouldn't happen with disjoint batches, but if it does,
  hand-decide and edit one of the `result-NN.yaml` files.
- The user dislikes a cluster: edit `synthetic_tags.yaml` directly. The
  renderer respects hand edits. Or edit `categories.yaml` and re-run the
  whole skill.
- All scripts only write to `<HUB_ROOT>/_index/`. Original `.md` files are
  never modified.
